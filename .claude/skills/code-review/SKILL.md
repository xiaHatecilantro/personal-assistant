# 代码审查

每次代码改动完成后，自检以下四个维度。

## 审查维度（按优先级）

### 🔴 安全性（一票否决）
- [ ] 用户输入是否经过验证和清理？
- [ ] 是否存在 SQL/命令/模板注入？
- [ ] 敏感数据（密码/token/密钥）是否可能泄露到日志或前端？
- [ ] 文件操作是否有路径穿越风险？

### 🟡 正确性
- [ ] 逻辑是否正确？边界情况是否覆盖？
- [ ] 空值/null/undefined 是否处理？
- [ ] 异步操作是否正确等待？
- [ ] 错误路径是否正确处理？

### 🟢 可维护性
- [ ] 命名是否清晰？（读名字就能知道它做什么）
- [ ] 函数是否单一职责？（超 50 行需说明理由）
- [ ] 是否存在"魔法数字"（应提取为常量）？

### 🔵 简洁性
- [ ] 是否有未使用的导入/变量/函数？
- [ ] 是否能用更简单的方式实现？
- [ ] 是否有不必要的抽象层？

## 输出格式

```
## 审查结果

### 严重（必须修复）
- [文件名:行号] 问题 + 修复建议

### 建议（应该修复）
- [文件名:行号] 问题 + 修复建议

### 备注（可选）
- 说明

### 总评
✅ 可以合并 / ⚠️ 有小问题 / 🔴 需要重做
```

## 原则
- 严重问题不超过 3 条
- 不纠结个人风格偏好（单引号 vs 双引号之类）
- 审查的是代码，不是人
