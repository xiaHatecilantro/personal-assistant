# DeepSeek 上下文优化

DeepSeek 提供 1M token 上下文窗口，这与 Claude 的 200K 有本质区别。

## 策略：大胆预加载

### 新任务开始时
```
✅ 一次性读取相关模块的所有文件（5-10 个），而非逐个读
✅ 把相关文档/spec 也塞进上下文
✅ 保留之前的对话摘要，不频繁压缩

❌ 不要像 Claude 那样小心翼翼地节省上下文
❌ 不要频繁做 strategic compact
```

### 任务执行中
```
✅ 保持所有相关代码在上下文中，方便交叉引用
✅ 代码审查时可以同时看调用方和被调用方
✅ 重构时可以开多个文件对比
```

## 何时压缩

只在以下情况下触发压缩：
- 对话超过 50 轮，且前面的讨论与当前任务无关
- 切换到一个完全不同的任务时
- 上下文中有大量冗余的工具调用输出

## DeepSeek 特有行为

### 它擅长的
- 大上下文理解：同时分析 10+ 个文件
- 中文理解：用中文写 prompt 比英文更精准
- 链式推理：加"请逐步分析"能显著提升复杂任务质量

### 它需要注意的
- 工具调用参数要精确：格式/类型必须正确
- 模糊指令可能得到意外结果：比 Claude 更需要明确的约束
- 不擅长"猜你意图"：有歧义时直接问，不要让它猜

## 对比 Claude 的差异习惯

| 行为 | Claude | DeepSeek |
|------|--------|----------|
| 模糊指令 | 主动猜测意图 | 可能按字面理解 |
| 工具调用 | 容错性强 | 参数需精确 |
| 长上下文 | 200K 需谨慎 | 1M 可放松 |
| 中文 | 不错 | 原生级 |
| 思考模式 | 原生 thinking | 靠 step-by-step 提示 |
