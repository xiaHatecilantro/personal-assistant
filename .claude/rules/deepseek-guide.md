# DeepSeek 使用指南

## 模型特性

| 特性 | DeepSeek v4 Flash | DeepSeek v4 Pro |
|------|-------------------|-----------------|
| 上下文 | 1M tokens | 1M tokens |
| 速度 | 快 | 中等 |
| 单价 | 低 | 中 |
| 适用 | 读取/搜索/简单编辑 | 设计/调试/复杂重构 |

## Prompt 技巧

### 1. 精确指令
DeepSeek 对模糊指令倾向于字面理解，不像 Claude 那样主动猜你意图。

```
✅ "在 src/auth.py 的 login 函数开头，添加参数校验：username 不能为空且长度 3-50"
❌ "给 login 加点验证"
```

### 2. 触发链式思考
对于复杂推理任务，加一句提示能显著提升质量。

```
✅ "请逐步分析这个 bug 的根因：{描述}"
✅ "先列出可能的方案，再逐一分析优缺点，最后推荐一个"
```

### 3. 工具调用参数
DeepSeek 对参数格式要求比 Claude 更严格。

```
✅ 文件路径用绝对路径："d:/self-program/src/main.py"
✅ 正则模式要精确："def\s+\w+\(.*\):" 而非 "def .*:"
❌ 避免空值参数，不用的参数直接省略
```

### 4. 中文交互
用中文写指令比英文更精准（DeepSeek 是中文原生模型）。

```
✅ "检查这段代码有没有 SQL 注入风险"
❌ "Can you check this code for SQL injection?" （理解可能不如中文准确）
```

## 上下文管理

### 与 Claude 的关键区别：别怕用满上下文

```
Claude Code 习惯 → 精简上下文，频繁 compact
DeepSeek 策略 → 充分利用 1M 窗口

操作上：
- 新任务：一次性读取 5-10 个相关文件，不用一个个读完再读下一个
- 代码审查：同时打开调用方和被调用方
- 调试：保留所有排查过程，方便回溯
```

### 何时才需要压缩
- 对话轮次超过 50，且前半部分与当前任务完全无关
- 切换到一个全新话题时
- 冗余的工具输出（如大量重复的 ls/cat）

## 已知限制

### 比 Claude 弱的地方
- **多步骤推理的连贯性**：复杂任务中途容易丢上下文，建议每个关键步骤重述当前目标
- **创造性生成**：设计 UI、起名等不如 Claude 有创意
- **容错性**：工具调用参数出错时，不如 Claude 善于自动修正

### 比 Claude 强的地方
- **中文理解和生成**：原生级别
- **大上下文理解**：1M 窗口不是摆设
- **代码生成**：对常见编程模式掌握很好
- **性价比**：同等能力下便宜很多

## 调试建议

当 DeepSeek 的输出不如预期时：

1. **是不是指令不够具体？** → 加约束条件
2. **是不是上下文太长忘记了？** → 重述关键信息
3. **是不是需要链式推理？** → 加"请逐步分析"
4. **是不是任务太抽象？** → 给具体示例
