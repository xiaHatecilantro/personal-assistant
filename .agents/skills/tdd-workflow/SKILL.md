# TDD 工作流

严格的 RED → GREEN → REFACTOR 循环。

## 循环

### 🔴 RED（先写测试，确认失败）
```
1. 写出你期望的函数/接口调用方式
2. 为成功情况、失败情况、边界情况各写一个测试
3. 运行测试 → 必须看到红色（测试失败）
   如果测试直接通过，说明：
   - 功能已经存在（不需要实现）
   - 测试写错了（没有真正测试新行为）
```

### 🟢 GREEN（最小实现，只求通过）
```
1. 写刚好够的代码让测试通过
2. 不要多写任何"以后可能用到"的代码
3. 不要重构（那是下一步的事）
4. 运行测试 → 绿色通过
```

### 🔵 REFACTOR（改善结构，保持绿色）
```
1. 消除重复
2. 改善命名
3. 提取公共逻辑
4. 每改一处就运行测试，保持绿色
5. 如果重构超过 5 分钟还没结束，停止并 commit
```

## 测试写法

```python
# ✅ 好的测试：描述行为
def test_validateToken_returns_false_for_expired_token():
    ...

# ❌ 差的测试：描述实现
def test_validateToken_calls_parseJWT():
    ...
```

## DeepSeek 注意
- 由于无原生 thinking 模式，写测试前先口头描述"我要测什么"
- 每次只写 1-2 个测试用例，不要批量写 10 个再实现
