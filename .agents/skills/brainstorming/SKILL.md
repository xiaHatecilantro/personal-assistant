# 需求澄清

当收到一个模糊的开发需求时，不要直接开始编码。先通过提问把需求弄清楚。

## 流程

### 1. 理解目标
先用自己的话复述一遍需求，确认理解正确：
> "我理解你的需求是：{你的理解}。对吗？"

### 2. 澄清关键点
逐一确认以下维度（挑相关的问，不要全问）：

| 维度 | 典型问题 |
|------|---------|
| 范围 | 这个功能是给谁用的？什么场景下用？ |
| 行为 | 正常情况做什么？异常情况（空数据/网络错误）做什么？ |
| 边界 | 数据量多大？有性能要求吗？ |
| 约束 | 有技术栈限制吗？需要兼容旧版本吗？ |
| 优先级 | 核心功能是什么？哪些可以后续迭代？ |

### 3. 提出方案（2-3 个选项）
如果存在多种实现方式，简要列出选项和 trade-off：
> "有两种方式：
> A: 方案A — 优点：简单快速；缺点：扩展性差
> B: 方案B — 优点：灵活可扩展；缺点：多写 50 行代码
> 推荐 A，因为当前需求不需要过度设计"

### 4. 确认后进入计划阶段
需求澄清完毕后，自动衔接 writing-plans 技能。

## 原则
- 每次最多问 3 个问题，不要审问式连问
- 不要为一个简单需求搞长篇分析
- 如果需求已经很明确（如"修 typo"），跳过此流程
